# Mongoose

## Mission

Mongoose is a Go SDK that turns idiomatic handler code into a production-ready application. Developers write the logic that's unique to their problem; mongoose provides the rest — CLI ergonomics, configuration, testing, mocking, sandboxing, and the small primitives that hold it all together. The project's commitments: spec-first development, Go idioms religiously, less code is more, supply-chain minimalism, and an A+ quality bar on every line that ships.

## Core directives

1. **Spec-first hard gate.** No code merges until a spec covering it is at status `accepted`. Specs live in [`specs/`](specs/). The gate is enforced by Otter (architecture). See [`specs/0000-spec-process.md`](specs/0000-spec-process.md) for the full process.
2. **Specs are the source of truth.** All public-facing documentation is generated by Magpie (docs & identity) from accepted specs. Never edit docs detached from a spec — the fix is to update the spec.
3. **Plans must be exhaustive.** Every plan presented for approval is fully resolved — no open questions, no decision points. Verbose is preferred over terse. Plans are executable contracts.
4. **Go idioms religiously.** `gofmt`, `go vet`, `staticcheck`. Small interfaces. Errors as values. No panics in library code. Context-first APIs.
5. **Less code is more.** Reuse before write. Three similar lines beats a premature abstraction. Don't add features, error paths, or abstractions beyond what the spec requires.
6. **Supply-chain minimalism.** `go.mod` stays small. Every direct dependency needs a written justification in the spec that introduced it. Prefer building a tiny in-tree convenience package over importing a library. Falcon (security) enforces.
7. **A+ bar.** Public API stable on first release. Zero-allocation hot paths where stated. Every exported symbol has a doc comment. Every package ships an example test.

A spec moves to `accepted` only when every required reviewer has signed off. Full matrix in [`specs/0000-spec-process.md`](specs/0000-spec-process.md).

|---|---|

## Before any change

1. Find the relevant accepted spec; if none exists, you're authoring one before you write code.
2. Read it. Match the diff to its `## API` and `## Test Matrix` sections.
3. If you're adding a direct dependency, fill in the spec's `## Dependency Justification` section and request Falcon review.
4. If you're changing the public surface (exported names, user-facing strings, CLI verbs), request Magpie review for naming and voice.
5. Run `gofmt`, `go vet`, `staticcheck`, and the test suite before opening a PR.

## Adding a dependency

Every direct dependency must answer these six questions in the spec that introduces it:

1. What does this give us that we can't write in <100 lines ourselves?
2. License compatible? (MIT, BSD, or Apache-2.0 acceptable; others require explicit Falcon sign-off.)
3. Last release date? (Stale > 18 months requires justification.)
4. Maintainer count? (Single-maintainer libraries require explicit risk acknowledgement.)
5. Vulnerability scan clean?
6. Pinned to a specific version? (No `latest`. No version ranges.)

If you cannot answer all six, do not add the dependency. Build the convenience in-tree.
