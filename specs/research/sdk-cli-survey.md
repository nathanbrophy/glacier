# SDK CLI Survey: Conventions Across Comparable Tools

## Public Summary

This survey catalogs how comparable developer-facing CLI tools (cargo, mage, task, goreleaser, golangci-lint, dagger, tilt, bun, deno, npm, pnpm, cargo-dist) shape the verbs that overlap with the Glacier SDK's nine kept commands: `init`, `new`, `generate`, `lint`, `test`, `version`, `completions`, `explain`, plus the `vibe` analogue. It records concrete grammar, flag names, output conventions, exit-code patterns, prompt design, telemetry posture, completion mechanics, and explanation surfaces. Each tool gets a short profile; every section ends with a "what works", "what does not", and "if I were doing this for Glacier" paragraph. The artifact's job is to ground future review and post-v0 amendments to spec 0032 in concrete prior art rather than taste alone.

## Mental Model

The cross-tool comparison frame uses three lenses.

**Lens 1: scaffold vs. surgical add.** `init` and `new` are different operations. `init` is bulk: create a directory tree, walk an interactive sequence, end on a "next steps" screen. `new` is surgical: edit an existing module, run codegen, leave the user in a working state. Tools that conflate the two (npm, deno, bun) leave users guessing whether `<tool> init my-app` writes into `./my-app/` or `./`. Tools that split them cleanly (cargo, dagger, glacier as locked) eliminate the ambiguity.

**Lens 2: thin wrapper vs. opinionated suite.** Some tools (mage, goreleaser, golangci-lint) wrap a single concern and expose configuration as their primary surface. Others (cargo, bun, deno) are opinionated suites: they run tests, format code, build artifacts, manage dependencies, and provide their own task runner. Glacier sits in the middle: opinionated about the development inner loop (lint, test, generate, scaffold) and intentionally silent about deploy, release, and dependency management.

**Lens 3: ceremony vs. zero-config.** The spectrum runs from `goreleaser init` (writes a 200-line YAML you must understand) to `bun init -y` (one keystroke, six files, you are running). Glacier's locked design biases toward zero-config (`glacier init --yes` produces a working CLI; defaults are coherent) while preserving an interactive path that walks the developer through choices for the first time.

```
                  ceremony                              zero-config
                  ────────────────────────────────────────────────
goreleaser init   X
cargo new                  X
glacier init                       X (interactive) X (--yes)
bun init                                                X
npm init                                                  X (-y)
```

Every recommendation below threads these three lenses.

## Per-tool surveys

### cargo (rustc 1.85, cargo 1.85, 2025)

`cargo` is the gravitational center for any Go-vs-Rust UX comparison. It owns nine of the ten verbs Glacier ships, and its conventions are familiar to most polyglot developers in the SDK's audience.

- **init / new.** `cargo init` works on the current directory; `cargo new <name>` creates a directory. The `--bin` (default) vs. `--lib` flag picks the template. `--name <pkg>` overrides the directory-name default. `--vcs git|hg|pijul|fossil|none` picks version control. `--edition 2024` picks Rust edition. There are no interactive prompts: every choice is a flag with a sensible default. License is *not* a built-in choice (you are expected to add `LICENSE` yourself).
- **generate.** No equivalent. `build.rs` plays this role; the interaction is implicit through `cargo build`.
- **lint.** No first-party command. `cargo clippy` is bundled but is technically a separate binary that ships in the toolchain. `cargo check` does compile-only.
- **test.** `cargo test [TESTNAME]` runs unit + integration + doc tests. Filtering by name is positional. `--release`, `--no-run`, `--quiet`, `--nocapture` (passed through `--`) are common flags. Output is line-oriented; no live panel; exit 101 on failure.
- **bench.** `cargo bench` is the canonical verb. Glacier's spec rejects this verb at top level and routes through `glacier test --bench`.
- **fmt.** `cargo fmt` is a thin wrapper around `rustfmt`. `--check` mode prints a unified diff and exits 1 on drift.
- **doc.** `cargo doc --open` builds and opens HTML docs.
- **completions.** `cargo`'s completion script is *not* generated by the binary; it ships with the toolchain at `$RUSTUP_HOME/toolchains/.../share/zsh/site-functions/_cargo`. `rustup completions <shell> [tool]` is the user-facing verb. This is a different model from kubectl-style "binary writes its own completion script."
- **version.** `cargo --version` and `cargo version` both work. Output: `cargo 1.85.0 (1d72e93b1 2025-01-13)`. There is no `--check` for upstream releases; `rustup update` plays that role.

**What works.** Stable, predictable verbs. Defaults that read like a tutorial. The `--quiet`, `-v`, `-vv` verbosity ladder is widely imitated. `cargo --help` returns a four-section grouped table (Build, Manifest, Package, Other) that scales as the surface grows.

**What does not.** The split between cargo, clippy, and rustfmt as nominally separate binaries adds onboarding friction (the docs constantly say "make sure you have rustup component add clippy"). License handling in `cargo new` is missing: most users want `--license MIT` and have to add it manually. The lack of a `cargo dev` umbrella creates a long tail of `cargo install <thing>` extensions whose discoverability is poor.

**If I were doing this for Glacier.** Glacier's locked design already does what cargo would do if it started over: license is a first-class flag and prompt; lint is bundled (no `glacier-lint` separate binary); the SDK ships completion writing in-binary. One piece worth borrowing post-v0: cargo's grouped four-bucket help table (CREATE / DEVELOP / INSPECT / UTILITY in the spec's D-S39) is the right pattern; the spec already adopts it.

### mage (mage v1.15, 2024)

mage is "make for Go": you write Go functions in `magefiles/`, mage compiles and runs them.

- **invocation.** `mage <Target> [args...]`. `mage -l` lists targets with their leading doc-comment line.
- **discovery.** mage uses Go reflection to discover exported functions in the magefiles package. Targets with `func(ctx context.Context) error` are async-cancellable.
- **completions.** None first-class. There is a community zsh completion in mage's repo; it shells out to `mage -l` and parses.
- **help.** `mage -h <Target>` prints the target's doc comment, with no flag introspection because targets are arbitrary Go.

**What works.** The doc-comment-driven help surface is the same pattern Glacier uses (markers in doc comments). mage proves users tolerate "your help text is your doc comment" as long as the doc comments are good.

**What does not.** No flag parsing inside targets: every flag is positional and stringly-typed. `mage -v` is verbose, but `-v` *also* might collide with a user target named `V`. Error messages on missing imports in magefiles are confusing: "package mage_output_template not found" is a real error users hit.

**If I were doing this for Glacier.** Borrow the doc-comment-as-help pattern (already locked in spec 0011 markers). Avoid mage's "no flag parsing" trap: every Glacier command has an explicit struct, not a free-form arglist. The SDK's `glacier <verb> --help` long form must be richer than mage's `-h <Target>`.

### task / Taskfile.dev (task v3.40, 2025)

Task is YAML-defined task runner with a Go binary. Its UX is the closest peer to "developer inner loop CLI" in the survey.

- **invocation.** `task <name>`, `task` (runs `default` task), `task --list` / `-l` (lists tasks with descriptions; tasks without `desc:` are hidden), `task --list-all` / `-a` (shows all).
- **execution modes.** `--watch` / `-w` reruns on source change (requires `sources:`). `--dry` compiles and prints commands without executing. `--parallel` / `-p` runs multiple tasks concurrently. `--force` / `-f` bypasses the up-to-date cache. `--summary <name>` shows dependencies, description, and commands without running.
- **interactive.** `--interactive` puts the user through prompts that the task definition declared; `prompt:` field on a task can require confirmation before running.
- **silent.** `--silent` / `-s` suppresses the `task: [name] <cmd>` echo line. By default Task prints what it is about to run, then the output of that command. This is the right default for a tool whose purpose is to make CI commands legible.
- **completions.** Hand-written scripts in `task/completion/` for bash, zsh, fish, pwsh. Distributed via the install method (Homebrew, Scoop, etc.), not generated by `task completions <shell>`.
- **help.** `task --help` lists all flags in a single ungrouped block.

**What works.** The default-task pattern (`task` with no args runs `default`). The `--list` / `--list-all` distinction (descriptions opt-in for visibility hygiene). `--summary` as a "show without run" verb. `--dry` as the canonical "show what would happen."

**What does not.** Completion shipped by package managers means new task verbs added to a project's Taskfile do not autocomplete: only the global flags do. Help is a flat wall.

**If I were doing this for Glacier.** Borrow `--dry-run` as the consistent name for "show without writing" (the spec already does this for `glacier new`). Borrow the principle that hidden things (commands without `desc:` in task) should never be the default: every Glacier command has a synopsis, full stop.

### goreleaser (v2.5, 2025)

goreleaser is the canonical Go release tool. Its UX is the cautionary tale for what happens when configuration grows without bounds.

- **commands.** `goreleaser init` writes an example `.goreleaser.yaml`. `goreleaser check` validates the config (and flags deprecations). `goreleaser build [--snapshot] [--single-target]` builds artifacts without releasing. `goreleaser release [--snapshot] [--skip=<phase>] [--clean]` runs the full release pipeline. `goreleaser healthcheck` validates that required external tools are on PATH. `goreleaser jsonschema` emits a JSON schema for editor integration.
- **config-first.** `.goreleaser.yaml` is the primary surface. The CLI is a thin set of verbs over a 1000+ line YAML schema. New users hit the wall on day one.
- **--snapshot.** Builds without git tag requirements; useful for testing the pipeline locally.
- **--skip.** Comma-separated phase list (`--skip=announce,publish`) lets users disable parts of the release.
- **deprecation surface.** `goreleaser check` prints deprecation warnings tied to spec versions in the YAML. This is the strongest UX choice in the tool.

**What works.** `init` writes a working starter config. `check` validates it. `jsonschema` for editor integration. Deprecation warnings are actionable and version-tagged.

**What does not.** The YAML grows without bounds. Errors regularly reference YAML paths users do not understand. The `--skip` flag's grammar (CSV) means typos are silent.

**If I were doing this for Glacier.** Avoid the goreleaser trap: keep config small, validated, layered (the spec does this). Borrow the `check` idea: `glacier generate --check` and `glacier lint` already play this role. Borrow the `jsonschema` emitter pattern post-v0 for `glacier explain --schema config` if config grows.

### golangci-lint (v1.62, 2025)

The reference for "many linters under one roof" UX in Go.

- **invocation.** `golangci-lint run [./...]`. Default is the cwd.
- **fix.** `--fix` enables in-place auto-fixes for linters that support it (gofumpt, goimports, etc.).
- **output formats.** `--out-format=colored-line-number|line-number|json|tab|checkstyle|code-climate|junit-xml|github-actions|sarif`. Multiple formats can be requested simultaneously. SARIF is the integration format for GitHub code scanning.
- **severity.** Severity is per-linter, not per-finding. The `severity` block in `.golangci.yml` lets users override per regex.
- **cache.** `~/.cache/golangci-lint/` stores per-file analysis results, keyed by content hash + analyzer fingerprint. Default TTL: indefinite; `--no-cache` disables.
- **config.** `.golangci.yml` (or `.toml`, `.json`) auto-discovered upward from cwd. `--config <path>` overrides. `--no-config` runs with defaults.
- **enable / disable.** `--enable=<linter>`, `--disable=<linter>`, `--enable-all`, `--disable-all`, `--default=<preset>` (none/standard/all/fast).
- **completions.** `golangci-lint completion bash|zsh|fish|powershell` emits a script to stdout (cobra-style).
- **version.** `golangci-lint --version`. There is no upstream-release check.

**What works.** SARIF output. The `--fix` flag is universally understood. Per-content-hash cache with implicit invalidation. Severity threshold as a flag (some linters have severity, some do not).

**What does not.** The config schema is long and requires multiple reads to internalize. Default is to enable a small set; users routinely want "fast preset" and have to learn the preset names. `--out-format` accepts multiple comma-separated values, but the CSV grammar is awkward when paired with `--out-format=junit-xml:report.xml,colored-line-number`.

**If I were doing this for Glacier.** The spec already mirrors the strongest choices: text/json/sarif outputs, content-hash cache, severity threshold flag, `--fix`. The lint cache lives at `<repo>/.glacier/lint-cache.json` (project-local, not user-cache). One missing pattern worth borrowing: a `--default=none|standard|all` preset key for the Glacier lint set, so users can opt out without enumerating six lints by name.

### dagger (v0.15, 2025)

dagger is the modular CI/CD tool. Its UX is the most opinionated about "everything is a sandboxed function."

- **commands.** `dagger init [--sdk=go|python|typescript|php|java] [--name=<name>]` initializes a Dagger module. `dagger develop` regenerates language bindings. `dagger call <function> [args]` invokes a function. `dagger install <module>` adds a module dependency. `dagger run <command>` runs a host command inside Dagger's session. `dagger login`, `dagger logout`, `dagger version`, `dagger --help`.
- **mental model.** Functions are sandboxed: no host filesystem access by default. Arguments are explicit. Functions chain via `|` (Dagger Shell).
- **modules.** Multi-language: a Go module can call a Python function and vice versa.
- **completions.** `dagger completion bash|zsh|fish|powershell`.
- **help.** `dagger call --help` introspects the *user's* functions and prints their flags. Live-introspected help is dagger's most beloved feature.

**What works.** Live help that introspects the user's own code. The `--sdk` flag for `init` is the right shape: language pluralism is a flag, not a default. Sandboxing as a default forces users into explicit-argument hygiene.

**What does not.** The mental model takes a tutorial to internalize. "Dagger Shell" is a new shell users have to learn. Errors that come from the engine's GraphQL layer are inscrutable when they leak through. `dagger run` and `dagger call` can both look right depending on context.

**If I were doing this for Glacier.** Glacier already commits to not introducing a new shell. The lesson worth borrowing: `dagger call --help` introspects user code; `glacier --help` should also surface user-defined commands once the SDK is being used to drive a project's own CLI. The SDK's spec already has this in the auto-injected version + completions subcommands.

### tilt (v0.33, 2025)

tilt is dev-environment orchestration. Worth surveying because of its TUI.

- **commands.** `tilt up [resource...]` runs the Tiltfile and opens a TUI/web dashboard. `tilt down` tears it all down. `tilt args [...]` updates positional args without restart. `tilt logs <resource>` streams resource logs. `tilt enable <resource>` / `tilt disable <resource>`.
- **TUI.** The terminal UI is keyboard-driven: arrow keys, enter to select, `?` for help. The web dashboard at `localhost:10350` is the primary surface.
- **completions.** `tilt completion bash|zsh|fish|powershell`.
- **help.** Standard cobra help; flat per-command flags.

**What works.** The TUI's `?` for help is the right keystroke. `tilt args -- --foo=bar` for argument forwarding without restart is a clever ergonomic.

**What does not.** The TUI is opinionated about colors and box-drawing in ways that fail on some Windows terminals. `tilt up` blocks the terminal; `tilt up --stream` is needed for CI use. The split between the TUI and the web dashboard means new users do not know which is canonical.

**If I were doing this for Glacier.** The locked `glacier vibe` is conceptually the closest peer to a tilt TUI moment, but tightly scoped: a 10 Hz frame loop, single-screen, exits on any keypress. Avoid tilt's web-dashboard split entirely; keep `glacier`'s TUI surface to the `vibe` command and the live status panels in `generate` and `test`. Borrow `?` as a help keystroke if `vibe` ever grows interactivity post-v0.

### bun (v1.2, 2025)

bun is the polished newcomer. UX choices are the strongest evidence for what "feels modern" to 2026 developers.

- **init.** `bun init [-y]` walks four prompts (project name, entry point, package type, Git init). `-y` accepts all defaults silently. The output is six files: `package.json`, `tsconfig.json`, `README.md`, `index.ts`, `bunfig.toml`, `.gitignore`. The whole flow is under 200 ms on a warm cache.
- **create.** `bun create <template> <dir>` clones from a curated template registry or any GitHub URL. This is bun's `cargo new` with template variety.
- **add / remove / update.** `bun add <pkg>`, `bun remove <pkg>`, `bun update [pkg]`. Symmetric verbs; deterministic exit codes.
- **run.** `bun run <script>` runs a `package.json` script. `bun <script>` (no `run`) is the shorthand. Bare `bun` with no args lists scripts.
- **test.** `bun test [pattern]`. Built-in test runner; jest-compatible API. `--watch` reruns on change. `--coverage` enables coverage. Output: per-file pass/fail with timing, plus a summary at the end.
- **completions.** Hand-written scripts shipped with the install. `bun completions bash` is *not* a verb (as of 1.2).
- **flags.** `--silent`, `--bun`, `--watch`, `--hot`, `--smol`, `--inspect`, `--cwd`, `--config`, `--help`. Help is grouped by section in the source but flat in `--help` output.
- **version.** `bun --version` and `bun -v` both work. Output is just the version: `1.2.0`.

**What works.** Sub-200-ms cold-start everywhere. `-y` as the universal "accept defaults" flag. Bare `bun` listing scripts is the right pun on bare `make`. Symmetric verbs (add/remove, install/uninstall as aliases). The package-script-vs-source-file resolution order is documented and stable.

**What does not.** Help is a flat wall in `--help`. Section grouping exists in docs but not in CLI output. No first-party completion subcommand as of 1.2. `bun -v` shows version, but `-V` is not aliased to `--verbose`; verbose is `--verbose` long-only.

**If I were doing this for Glacier.** Borrow the sub-200-ms cold-start budget religiously (the spec sets `glacier init` at 200 ms total wall, which matches). Borrow `-y` as `--yes` (already locked). Borrow the symmetric verb pattern (`init` / `new` as the spec has it). Differentiate from bun on completions: ship `glacier completions <shell>` as a first-class verb (already locked). The grouped help format the spec mandates is a strict improvement over bun's flat wall.

### deno (v2.1, 2025)

deno is bun's older sibling. Its UX deserves separate study because of its permissions model and its `deno fmt` / `deno lint` / `deno test` first-party tooling.

- **commands.** `deno init [--lib|--serve] [name]`. `deno run [--allow-read=path] script.ts`. `deno test [--coverage] [--filter] [pattern]`. `deno fmt [--check]`. `deno lint [--rules] [--rules-tags]`. `deno bench`. `deno doc [--html|--lint]`. `deno completions <shell>`. `deno upgrade`. `deno info <module>`.
- **permissions.** `--allow-read`, `--allow-write`, `--allow-net`, `--allow-env`, etc. The default is deny; users opt into capabilities per invocation. This is the strongest security UX in the survey.
- **fmt vs. lint.** Two separate verbs, each with `--check` (CI mode) and a default in-place mode (write). Output is unified-diff on `--check` failure.
- **test.** `deno test --coverage=cov_profile` writes coverage to a directory; a separate `deno coverage cov_profile` post-processes. Two-stage UX is unusual and frequently confuses CI authors.
- **completions.** `deno completions bash|zsh|fish|powershell`. First-class.
- **upgrade.** `deno upgrade` self-updates. The Glacier spec explicitly rejects this (D-S26).
- **doc.** `deno doc --html ./...` builds HTML docs and serves them. Glacier defers `glacier doc` per the spec.

**What works.** Per-verb design that is unmistakable. `--check` as the CI-mode flag for both `fmt` and `lint`. Permissions as deny-by-default. `deno info <module>` for explaining what would be downloaded.

**What does not.** `deno upgrade` mutates the user's binary; users on locked-down corporate machines find it confusing or scary. Two-stage coverage workflow (`deno test --coverage=DIR` then `deno coverage DIR`) is a wart inherited from V8's coverage tooling. `deno doc` mixes "lint my doc comments" and "build HTML site" under one verb with a flag.

**If I were doing this for Glacier.** The locked self-update refusal (D-S26) is the right call: explicitly *not* deno-style. Borrow `--check` as the consistent CI-mode name for both lint and generate (the spec does this). Avoid deno's two-stage coverage UX: `glacier test --cover` should produce the merged report in one invocation (the spec mandates this).

### npm (v11, 2025) and pnpm (v9, 2025)

The de-facto baselines for "developer reaches for a CLI" in JS/TS land. Listed together because their conventions are 90% shared and divergence is interesting.

- **init.** `npm init [<initializer>]`. Without initializer: walks 10+ prompts. `npm init -y` skips them. With initializer: `npm init react-app foo` is a shorthand for `npx create-react-app foo`. This conflation between "init my package.json" and "scaffold a project from a template" is npm's most-confused UX.
- **install / add.** `npm install <pkg>`, `pnpm add <pkg>`. pnpm's choice to use `add` (matching cargo / bun) is widely seen as more legible.
- **run.** `npm run <script>` runs a `package.json` script. Bare `npm run` lists scripts.
- **test.** `npm test` is an alias for `npm run test`. There is no first-party test runner.
- **completions.** `npm completion` writes a bash/zsh script to stdout. There is no `--shell <name>` selector; npm autodetects the shell from `$SHELL`. pnpm uses `pnpm completion <shell>`.
- **version.** `npm --version` prints the npm version. `npm version` is a different command that bumps `package.json` version. This naming collision is the worst in the survey.
- **explain.** `npm explain <pkg>` prints the dependency path that brought a package in. This is the closest analogue to `glacier explain` in any surveyed tool.
- **help.** `npm help <command>` opens a man page if available, else falls back to a usage string. This works on Linux/macOS and falls back awkwardly on Windows.

**What works.** `npm explain` as a first-class verb is rare and beloved. `pnpm`'s `add` / `remove` pair (vs. npm's `install` / `uninstall`) is more symmetric. Bare `npm run` listing scripts.

**What does not.** `npm version` vs. `npm --version` is the textbook example of a CLI naming collision. `npm init <initializer>` overloading two distinct operations on one verb is the textbook example of a CLI verb overload. man-page-based help that does not work on Windows.

**If I were doing this for Glacier.** Avoid the `version` collision by reserving `--version` (and `-v`) for "print version" *and* having `version` be the subcommand for the same purpose with extras (`--check`, `--json`). The spec already does this: `-v` is `--version`, the `version` subcommand adds `--check` and `--strict` and `--json`. Borrow `npm explain`'s name and its model: `glacier explain <topic>` mirrors it directly (already locked).

### cargo-dist (v0.27, 2025)

cargo-dist is the most modern release-tooling experience. Worth surveying for its `init` flow specifically.

- **init.** `cargo dist init` walks an interactive flow that asks: which platforms to build for, which formats (tarball, msi, dmg), CI provider (GitHub / GitLab), installer type (curl-pipe-bash / homebrew / msi). It writes `Cargo.toml` `[workspace.metadata.dist]` keys and `.github/workflows/release.yml`.
- **plan.** `cargo dist plan` tells you what would be built and where without doing it. The output is a markdown table.
- **build.** `cargo dist build` produces artifacts locally.
- **manifest.** `cargo dist manifest` describes the artifacts of a tagged release.
- **generate.** `cargo dist generate-ci` rewrites the CI YAML from current config.
- **selfupdate.** Disabled by default; opt-in via config.

**What works.** `init` as a guided, opinionated walk. `plan` as a "show me" verb (the right name). `manifest` for downstream tools. The opt-out-by-default selfupdate posture is closer to Glacier's stance than deno's opt-in.

**What does not.** The CI-rewrite mechanism is invisible: users edit the workflow, run `init` again, lose their edits. Without a `--dry-run` mode for `init`, accidental rewrites are common. The output of `plan` is a markdown table that does not render in some terminals.

**If I were doing this for Glacier.** Borrow `plan`-as-a-name for the principle, but the spec's existing `--dry-run` (on `glacier new`) covers the same need without introducing a new verb. The cargo-dist lesson worth carrying forward: when `init` rewrites files the user has touched, refuse without `--force` and provide a diff (the spec does this for `glacier init`).

## Adjacent reference points (briefer treatment)

Five additional tools are worth a paragraph each because they inform specific Glacier choices without warranting full sections.

**gh (GitHub CLI, v2.65, 2025).** `gh` ships `gh extension`, `gh completion bash|zsh|fish|powershell`, `gh auth login` (interactive), `gh repo create` (interactive scaffold). The strongest UX choice: `gh` checks for updates daily, prints a one-liner at the bottom of relevant commands ("A new release of gh is available: 2.65.0 -> 2.66.0"), and the check is opt-out via `GH_NO_UPDATE_NOTIFIER`. Glacier's `version --check` is more conservative (user must run the verb to trigger the check) but the daily-check pattern is the right post-v0 amendment if users complain that `version --check` is too manual. The `gh repo create` interactive flow is the closest peer to `glacier init`'s prompt sequence; both tools converge on "skip with `--yes` (or `gh repo create <name> --public`) for non-interactive use."

**kubectl (v1.32, 2025).** kubectl's help system is the gold standard for "live introspected help over a discovered surface." `kubectl get --help` lists every resource type the cluster exposes (pods, services, deployments, plus CRDs). Glacier's `glacier --help` after `glaciergen` runs in a user's project should feel similar: the registered command tree drives the help surface. kubectl also ships `--dry-run=client|server|none`, which is the strongest evidence that `--dry-run` should be a value-flag rather than a boolean once the SDK gains a server-side dry-run analogue (post-v0).

**hugo (v0.140, 2025).** hugo is the precedent for "scaffold a site, watch for changes, regenerate." `hugo new site <name>`, `hugo new <content-path>`, `hugo server -D`, `hugo --gc`, `hugo completion bash`. The `hugo new <content-path>` verb is a direct peer to `glacier new <subverb>`; both follow the surgical-add pattern. hugo's rich `--help` per command (with examples) is a strong precedent for the spec's per-command long-help format (D-S40).

**ripgrep (v14.1, 2025).** ripgrep is included for one reason: the `--no-config`, `RIPGREP_CONFIG_PATH`, `--debug`, `--type-list`, `--type-add` ergonomic ladder. ripgrep's sub-100-ms startup is a public commitment; the team rejects features that compromise it. Glacier's per-command cold-start budgets in spec 0032 (30 to 50 ms) put it in the same neighborhood. ripgrep's choice to honor `--threads N` plus `--threads 0` (auto) is the model for `glacier generate --parallel N` plus `--parallel 0` (auto = `min(NumCPU(), 8)`).

**fd (v10, 2025).** fd is included for its color-and-glyph defaults. fd uses ANSI color tokens for "directory", "executable", "symlink" by default and respects `LS_COLORS`. The pattern Glacier adopts (`term.*` palette tokens per spec 0001) is structurally identical: every output color resolves through a named token, never a hex code in user-facing code.

## Cross-tool patterns worth borrowing

The strongest converging patterns across the surveyed tools.

**1. `--check` is the universal CI-mode flag.** cargo fmt, deno fmt, deno lint, golangci-lint (via `--out-format=...` and exit codes), goreleaser check. The Glacier spec uses it for `glacier generate --check` and effectively for `glacier lint` (lint is implicitly check-mode). The semantic: zero exit if clean, non-zero if drift, no in-place mutation.

**2. `--dry-run` is the universal "show what would happen" flag.** task --dry, kubectl --dry-run=client, terraform plan (verb), cargo-dist plan (verb), Glacier `new --dry-run`. The semantic: render the action plan, write nothing, exit 0.

**3. `-y` / `--yes` skips all prompts and accepts defaults.** apt-get -y, npm init -y, bun init -y, Glacier (locked: `glacier init --yes`). When defaults are coherent this is the single most-loved flag in any CLI.

**4. Symmetric add/remove verbs.** cargo add / cargo remove. bun add / bun remove. pnpm add / pnpm remove. Glacier's `new` subcommands (`new package`, `new command`, `new option`) follow this pattern; a future `remove` verb is documented but deferred.

**5. Layered config: defaults < file < env < flags.** golangci-lint, dagger, goreleaser, deno. Glacier's `conf` package lands here directly. Spec D-S21 puts `GLACIER__*` env vars in the right tier (between file and flags).

**6. Output format flag with text/json/sarif/junit options.** golangci-lint, deno test, go test (-json). Glacier `lint` and `test` both support text/json/sarif (lint adds sarif, test adds junit). SARIF in particular is the integration format with GitHub code scanning.

**7. Shell completion as a first-class subcommand.** kubectl, helm, hugo, cobra-based tools, dagger, deno, pnpm, golangci-lint, go (`go completion -h`). The format `<tool> completions <shell>` is the dominant pattern. Glacier locks this exactly.

**8. `explain` as a verb for stable cross-references.** rustc --explain (compiler error codes), npm explain, golangci-lint help <linter>. Glacier `explain` covers markers, exit codes, and config keys. The pattern's strength is *stability*: an exit code that says "run `<tool> explain 66`" is friendlier than a long inline paragraph and ages better.

**9. Per-content-hash result cache.** golangci-lint, go test (test cache), deno cache, npm (registry cache). Glacier's `<repo>/.glacier/lint-cache.json` follows this pattern. The key design choice is content-hash, not mtime: mtime-based caches break on git checkout because mtimes change but content does not.

**10. Live introspected help.** dagger call --help (introspects user functions), kubectl --help (introspects discovered API resources), Glacier `glacier --help` (after `glaciergen` runs, the help reflects the user's command tree). This is what makes a CLI feel like it belongs to the user, not the framework.

**11. Sub-200-ms cold start.** bun, deno, fd, ripgrep. Tools that start fast feel different. The Glacier spec sets per-command cold-start budgets at 30 to 50 ms.

**12. NO_COLOR plus tool-specific override.** NO_COLOR is the cross-tool standard (no-color.org); GLACIER_NO_COLOR is the tool-specific override. cargo, golangci-lint, ripgrep all honor NO_COLOR. Glacier honors both per spec 0001 D45 and 0032.

**13. Build-info fallback for version strings.** Go's `runtime/debug.ReadBuildInfo` gives a usable version string for `go run`-built binaries. Glacier's `version` falls back to this when `-ldflags` injection is absent. cargo, goreleaser, and deno all have analogous fallbacks (cargo via `CARGO_PKG_VERSION` macro; goreleaser injects via `-ldflags` only).

**14. Self-update refusal as a stance.** cargo (no self-update; rustup is separate), goreleaser (no self-update), Glacier (D-S26). The opposite (`deno upgrade`, `bun upgrade`) is the loud minority. The argument for refusal is that locked-down environments and supply-chain auditors require external installation.

**15. Telemetry opt-in via standard env vars.** OTEL_EXPORTER_OTLP_ENDPOINT (OpenTelemetry standard) is the right env var to honor. dagger uses it. Glacier honors it (D-S25). Tools that invent their own telemetry env var (e.g. `<TOOL>_TELEMETRY=on`) fragment the ecosystem.

## Cross-tool failures worth not borrowing

The patterns the surveyed tools got wrong, listed so the SDK does not repeat them.

**1. Verb overload between version-print and version-mutate.** `npm version` *bumps* `package.json` version; `npm --version` *prints* npm's version. Glacier dodges this: `glacier version` prints; `glacier --version` and `-v` print; nothing in the SDK mutates a version. Keep this rule.

**2. Self-mutating binaries.** `deno upgrade`, `rustup self update`, `bun upgrade`. Locked-down environments break; supply chains get harder to audit; users on Homebrew and `go install` already have an upgrade story. Refuse permanently (spec D-S26 does this).

**3. `init <thing>` overloaded as `init my-package` *and* `init my-template`.** npm init does this; bun create separates it; cargo new is unambiguous. Glacier's clean `init` (scaffold a project) vs. `new` (add to a project) split is the right call.

**4. CSV grammar for multi-value flags.** goreleaser `--skip=announce,publish`, golangci-lint `--out-format=json,sarif`. CSV typos are silent. Glacier should prefer repeated `--only=cli --only=mock` over `--only=cli,mock` when introducing multi-value flags. (The spec uses `[]string` with a `+glacier:choices` constraint per value; the codegen should treat repeated invocations as the canonical form.)

**5. Two-stage coverage workflow.** `deno test --coverage=DIR && deno coverage DIR`. Two verbs to run a single intent. Glacier `test --cover` should produce the merged report in one shot (the spec mandates this).

**6. Help text as a flat wall.** bun, mage, raw cobra default. After ten verbs, the wall scrolls. Glacier's spec mandates four grouped buckets (CREATE / DEVELOP / INSPECT / UTILITY) per D-S39 and per-command long help per D-S40. This is a strict improvement.

**7. Hidden commands by default.** task hides commands without `desc:`. golangci-lint has linters that are not enabled by default but live in the binary. Glacier should make every command visible in `--help` and every flag listed in the per-command long help. No hidden surface.

**8. Completion shipped only by the package manager.** task, mage, cargo. New verbs added by upgrades or by the user's project do not autocomplete until the package-manager-shipped script is updated. Glacier ships `glacier completions <shell>` so the script reflects the *running* binary's command tree (D-S34).

**9. Telemetry that phones home by default.** Earlier versions of next.js, create-react-app, and dotnet did this. The community backlash was severe. Glacier refuses: zero SDK telemetry, OTEL only when the user sets the standard env var.

**10. man-page-based help on Windows.** npm's `npm help <command>` opens a man page; on Windows it falls back to a usage line. Mixed UX. Glacier's `glacier <verb> --help` is a stdout-rendered block, terminal-agnostic.

**11. Inscrutable engine errors.** dagger's GraphQL layer leaks errors users cannot map back to their input. Glacier's `sdkerr` package (D-S7) exists specifically to wrap *every* boundary error in a sentinel or typed value with an actionable hint and an `glacier explain <code>` link.

**12. The `goreleaser` YAML cliff.** A 1000-line config schema that grows over time. Glacier keeps configuration under 10 keys (spec D-S21) and routes complexity into per-command flags + markers, not into a config file.

**13. Mixed canonical surfaces.** tilt's TUI vs. web dashboard split. Glacier ships `glacier vibe` as a terminal-only experience and refuses to add a web surface in v0.

**14. Verbosity flag conflicts with version flag.** Some tools use `-v` for both verbose and version, causing per-invocation ambiguity. Glacier's spec D-S31 reserves `-v` for `--version` and uses `-V` / `--verbose` and `--very-verbose` (long-only) for verbosity, matching cargo's `-v` / `-vv` *content* (verbose ladder) but resolving the version collision explicitly.

**15. Ungrouped flag tables that grow forever.** bun, mage, goreleaser. Help becomes unscanable. Glacier groups flags into per-command sections and surfaces global flags via a "GLOBAL FLAGS" footer (D-S39).

## Per-command recommendations

For each Glacier command, the recommendations cite the tools that informed the design and give a focused "what's locked, what to revisit post-v0" note. Recommendations do not relitigate locked decisions; they identify which surveyed conventions reinforce the lock and which adjacent ideas might inform a future amendment.

### vibe

**Informed by.** tilt's TUI; cargo's lack of a "delight" command (a gap, not a model).

**What's locked.** 10 Hz frame loop. Polar bear plus wordmark. Tip rotation every 5 s. Press any key to exit. ASCII fallback. `--seed` for deterministic ordering.

**Cross-tool reinforcement.** No surveyed tool ships a `vibe` analogue. The closest peer is `cowsay` shipped by some package managers. The reinforcement direction: a tool's *flagship dogfood of its own animation primitive* is rare enough that vibe carries weight as a demonstration of `term.Animator` and its slog-handler interception.

**Post-v0 watch.** If `vibe` ever grows interactivity (arrow keys to switch tips, `?` for help) borrow tilt's TUI keystroke conventions: `?` for help, arrow keys for selection, `q` and Esc for quit alongside any-key.

### version

**Informed by.** cargo (`cargo --version` output shape), bun (`-v` short), npm (the cautionary tale: `version` collision), deno (build-info fallback patterns), golangci-lint (no upstream check; the absence is the signal).

**What's locked.** `glacier version` prints; `--check` adds GitHub Releases lookup; `--json` for machine consumers; `--strict` for CI; `-ldflags` injection with `runtime/debug.ReadBuildInfo` fallback; cache at `<UserCacheDir>/glacier/versioncheck.json` with 24 h TTL; advisory flock for write arbitration.

**Cross-tool reinforcement.** The spec's three-key version output (build time, go toolchain, os/arch) matches cargo's `cargo 1.85.0 (1d72e93b1 2025-01-13)` density without the trailing parenthetical that cargo's users find inscrutable. The `--check` pattern with TTL cache is novel in the Go-CLI space; closest peer is `gh` (GitHub CLI), which checks for updates daily and prints a one-liner. The `--strict` flag (network failure means exit 68) is the cleanest CI escape hatch.

**Post-v0 watch.** Watch for "I am behind several versions; show me the changelog" requests. The spec already prints a `go install` line; a future amendment might add `--changelog` that fetches release-notes markdown, but the SDK's current refusal to grow this surface is the right call until v0 ships.

### generate

**Informed by.** kubectl `kustomize`, gh `gh extension`, golangci-lint (multi-tool umbrella with content-hash cache), dagger develop (regenerates language bindings).

**What's locked.** Three generators (cli, mock, httpmock) under one umbrella. `--check` for drift (exit 69). `--only=cli|mock|httpmock`. Repeated-flag form for multi-value (`--only=cli --only=mock`) per spec 0011. Per-generator status panel rows. Unified diff blocks on drift.

**Cross-tool reinforcement.** golangci-lint is the canonical "many tools under one verb" precedent; `--out-format=json` and `--out-format=sarif` are not yet surfaced on `generate` (lint and test have them) and probably should not be: generate's job is to write or detect drift, not produce machine-readable findings.

**Post-v0 watch.** If a fourth generator lands (e.g. an OpenAPI generator), the registry pattern in the spec (`name`, `fn` pairs in `generate.go`) extends without ceremony. Watch for the "I want generate to *also* run go generate" request: keep this out of scope; users compose `glacier generate && go generate` themselves.

### lint

**Informed by.** golangci-lint (everything), deno fmt + deno lint (the `--check` mode), cargo clippy (the bundling lesson: bundle, do not split into a separate binary).

**What's locked.** gofmt + go vet + staticcheck + six Glacier-specific lints. `--fix` for auto-fixable findings. `--severity=error|warning|info` threshold. `--format=text|json|sarif`. Content-hash cache at `<repo>/.glacier/lint-cache.json`. Exit 65 on findings ≥ severity.

**Cross-tool reinforcement.** SARIF support matches GitHub code scanning (and Azure DevOps, Sonar). The `--severity` threshold flag is the right level of granularity: per-finding regex configuration (golangci-lint's `severity:` block) is over-engineering for v0. The cache layout matches golangci-lint's content-hash discipline; Glacier's choice to put the cache at `<repo>/.glacier/` (project-local) instead of `~/.cache/glacier/` (user-global) is correct because it makes CI-runner caches deterministic and reproducible per branch.

**Post-v0 watch.** Watch for "make a Glacier lint preset" requests (a la golangci-lint's `--default=fast`). The spec's six lints are already a curated set; a `--preset=none|standard|all` flag is the right shape if the lint set grows past 12.

### test

**Informed by.** go test (the wrapped subprocess), bun test (live UI; jest-compat), cargo test (filtering grammar), deno test (two-stage coverage as anti-pattern), golangci-lint (sarif output as integration story), benchstat (the right tool for `--bench`).

**What's locked.** Wraps `go test -json`. Live status panel with rolling rows. Summary block with slowest tests, coverage vs. threshold, failure block with isolation invocation. `--bench=<re>` with `--update-baseline`. JUnit, SARIF, JSON output formats. Exit 66 on failure or > 5% bench regression.

**Cross-tool reinforcement.** The slowest-tests block in the summary is rare; `cargo test` lacks it; `go test` lacks it; pytest and jest both have it. The "isolation invocation" hint at the bottom of the failure block (`glacier test ./conf/ -run TestLayerConflict_File_vs_Env -v`) is the single most actionable thing a test runner can print on failure; pytest does the equivalent.

**Post-v0 watch.** Watch for "I want gradual coverage thresholds per package" requests. The spec's single threshold is the right v0 simplification; multi-threshold support is a config-file growth path, not a flag.

### init

**Informed by.** cargo new / cargo init (the split), bun init -y (the keystroke-minimal flow), cargo-dist init (the interactive walk), goreleaser init (the cautionary tale), deno init (the `--lib|--serve` flag pattern).

**What's locked.** `glacier init [dir]` with module-path / template / license / mascot / git-init prompts. `--yes` skips all. `--force` overrides existing files. Three templates: `library-only`, `cli-app`, `both`. License flag with SPDX values. Mascot is a curated 6-item choice.

**Cross-tool reinforcement.** Glacier's prompt sequence is the union of bun's (4 prompts) and cargo's (zero prompts). The "next steps" final screen is rare and beloved; bun and cargo-dist both ship it; goreleaser does not. The mascot prompt is novel; closest precedent is `gh repo create` template selection.

**Post-v0 watch.** Watch for "create a project from a custom template URL" requests. bun's `bun create <template>` model is the obvious extension. Resist for v0: the curated three templates is the right surface area.

### new

**Informed by.** cargo (no peer), rails generate (`rails g controller Foo`; the surgical-add precedent), gh extension create (a different surgical add), nest g (NestJS CLI; the most polished surgical-add UX in any framework).

**What's locked.** `new package`, `new command`, `new option` as subverbs. `--dry-run` for preview. AST-edit existing files via `go/ast` + `go/format`. Auto-rerun cligen after writes.

**Cross-tool reinforcement.** Rails `rails generate` is the strongest peer; its scaffold subcommand (`rails g scaffold User name:string`) is the model for Glacier's grammar (typed positional + flags). NestJS `nest g` (e.g. `nest g resource users`) is the closest in terms of surgical AST edits. Both tools converge on `--dry-run` as the universal preview flag and both update parent files (Rails: routes.rb; NestJS: module.ts; Glacier: zz_generated_cli.go).

**Post-v0 watch.** Watch for "new test" requests (scaffold a test file for an existing function). The spec's three subcommands (package / command / option) cover Glacier's primary extension points; a fourth `new test` is plausible when fixture/mock packages stabilize.

### completions

**Informed by.** kubectl completion (cobra-style), helm completion, golangci-lint completion, deno completions, dagger completion, gh completion. The dominant convention.

**What's locked.** `glacier completions <shell>` with shell in {bash, zsh, fish, pwsh}. Output to stdout. Walks `cli.Default`'s registered command tree at runtime (so the script reflects the *running* binary's tree).

**Cross-tool reinforcement.** Every cobra-based CLI follows this pattern. The Glacier spec's choice to write the script in stdlib Go (no cobra) means the script generator is in `cli/gen` and ships per-shell writers; this is more code than cobra's reflection-based generator but stays within the supply-chain minimalism rule.

**Post-v0 watch.** Watch for nushell support requests. nushell's completion model is different (declarative). It is the right "first new shell" to add when a fifth shell is justified.

### explain

**Informed by.** npm explain (the verb name), rustc --explain (the cross-reference precedent for compiler error codes), golangci-lint help <linter> (linter explanation pattern), `man <command>` (the deepest precedent: a stable second surface for documentation).

**What's locked.** `glacier explain <topic>` where topic is a marker, exit code, or config key. `--list` enumerates all topics. `embed.FS` of pre-rendered topic files generated at build time from spec sections. CI byte-equality check enforces freshness.

**Cross-tool reinforcement.** npm explain is the only cross-tool peer with the same verb name; its model (explain a *thing in your project*, not the tool's own surface) is different but the verb's *fit* is identical. rustc `--explain E0308` is the model for cross-referencing exit codes from error output (`Run \`glacier explain 66\`` is the direct analogue of rustc's "for more information about this error, try `rustc --explain E0308`"). The spec-extracted topic files are a strong UX choice: every documented exit code, marker, and config key is reachable from the CLI without a network round-trip.

**Post-v0 watch.** Watch for "explain how to fix this lint finding" requests. The spec already has `glacier explain <topic>` for stable IDs; a per-lint explain (`glacier explain exported-doc-comment`) is a low-cost extension that fits the existing surface.

## FAQ

**Q: Why no top-level `glacier fmt`?**
A: spec 0032 D-S9 documents the rejection. Marker normalization belongs inside `glaciergen --normalize`, not as a separate verb. The cross-tool precedent (cargo fmt, deno fmt, gofmt itself) suggests a `fmt` verb is conventional, but Glacier's actual fmt-able surface (markers in doc comments) is too narrow to justify a top-level verb. If a future Glacier-style fmt grows beyond markers (e.g. to standardize doc-comment structure), revisit.

**Q: Why no `glacier upgrade`?**
A: spec 0032 D-S9 and D-S26 document the refusal. Self-mutating binaries break locked-down environments (corporate workstations, CI runners with read-only PATH dirs) and complicate supply-chain audits. The `go install` line printed by `version --check` is the upgrade story. cargo, goreleaser, and gh all converge on the same stance.

**Q: Why is `bench` an alias for `test --bench` instead of a top-level verb?**
A: spec 0032 D-S9 documents the alias-only choice. cargo bench is a top-level verb in cargo because Rust's bench infrastructure is separate from test; in Go, `go test -bench` is the canonical invocation. Mirroring `go test -bench` as `glacier test --bench` keeps the Go user's mental model intact.

**Q: Why no `glacier ask` or `glacier chat`?**
A: spec 0032 D-S9 documents the permanent rejection. LLM-shell verbs imply network calls, API keys, and a privacy posture incompatible with the SDK's "never phones home" rule. If users want an LLM helper, they install one separately.

**Q: How does the SDK feel different from cargo?**
A: cargo is a Rust-tool; the SDK is a Go-tool. The grammar mirrors cargo's where the conventions are language-neutral (`init`, `new`, `test`, `completions`) and diverges where Go's conventions differ (`generate` is a Go-codegen umbrella; `lint` bundles staticcheck + go vet; `bench` is a flag, not a verb; the version output uses `runtime/debug.ReadBuildInfo` fallback for `go run` builds). Where cargo splits (clippy, rustfmt as separate binaries), Glacier bundles. Where cargo prompts zero, Glacier prompts when it serves the user.

**Q: How does the SDK feel different from bun?**
A: bun's surface is large (test runner, package manager, runtime, bundler) and its primary delight is sub-200-ms startup. The SDK's surface is narrow (nine verbs, no package management, no runtime, no bundler) and its primary delight is "every Glacier package is exercised by at least one verb." bun init is faster than `glacier init` and has fewer prompts; `glacier init` walks more choices because mascot, license, and template are first-class. bun has no `explain` verb; Glacier does.

**Q: Why is `--check` the consistent CI-mode flag?**
A: cargo fmt --check, deno fmt --check, deno lint --check, goreleaser check, kubectl --dry-run=client. The convention is universal across modern tools. The semantic is unambiguous: zero exit if clean, non-zero if drift, no in-place mutation. Glacier uses `--check` on `generate` (drift detection) and the lint suite is implicitly check-only (always reports; never mutates without `--fix`).

**Q: Why does `--dry-run` exist on `new` and not on `init`?**
A: `init` writes to a target directory and refuses on collision without `--force`; the user can preview by running on a temp directory. `new` writes into an *existing* module, often touching files the user has hand-edited; previewing is the high-leverage feature. The asymmetry tracks the surgical-vs-bulk distinction in the mental-model section.

**Q: Why does the SDK not honor `XDG_CONFIG_HOME` and `XDG_CACHE_HOME` directly?**
A: spec 0032 routes through stdlib `os.UserConfigDir` and `os.UserCacheDir`, which honor the XDG variables on Linux and pick the platform-correct directory on macOS and Windows. The behavior is XDG-compliant on Linux without the SDK explicitly reading the env vars.

**Q: Why are there only six mascots?**
A: spec 0032 D-S38 sets the curated library at six. The number balances "user has a real choice" against "Magpie can author six in detail." A larger library would dilute the per-mascot polish; a smaller library (one) would feel over-prescriptive. The pattern matches GitHub Octicons' approach to a curated brand-icon library.

**Q: Why does `glacier vibe` exist at all?**
A: It is the SDK's flagship dogfood of `term.Animator`. Every framework component (the animator's slog-handler interception, the gradient renderer, the figgen banner, the kaomoji palette, the signal handling, the raw-mode discipline) is exercised in one verb. If `vibe` works, the framework works. The cross-tool precedent is thin (cowsay, sl, lolcat) but the engineering precedent is deep: every successful framework ships a `hello-world`-style sample that proves the rest of the framework works. `vibe` is the SDK's `hello-world`.

**Q: Why does `version` cache for 24 hours?**
A: 24 hours is the de-facto modern interval (homebrew checks daily, gh checks daily, asdf checks daily). Shorter intervals waste GitHub API rate limits and create flicker in CI logs. Longer intervals (a week) feel stale. The spec lets users override via `versioncheck.cache_ttl`.

**Q: How would a fourth generator (say, OpenAPI) plug into `generate`?**
A: spec 0032's generator registry is an explicit `[]struct{ Name string; Fn GenFn }` list in `generate.go`. Adding a row with `name: "openapi", fn: openapi.Generate` is the change. The status panel, drift mode, `--only=openapi` selection, and exit codes inherit automatically. No new top-level verb needed; the umbrella scales by extension rather than by verb proliferation.

**Q: Why does `lint` include a `naked-any` lint as opt-in?**
A: spec 0032 lists six Glacier-specific lints; `naked-any` is opt-in (warning level, off by default). The pattern matches golangci-lint's per-linter `enable:` config: a lint that catches a stylistic divergence (using `any` where a more specific generic constraint would do) is valuable but generates noise in legacy code; opt-in is the right default. Users enable it once their codebase is generic-clean.

**Q: Why doesn't `test` have a `--watch` flag?**
A: spec 0032 does not surface `--watch`. The cross-tool precedent (bun test --watch, deno test --watch, jest --watch) is strong. The decision to defer for v0 is a scope cap, not a stance: `--watch` requires fsnotify (or a polling fallback) and a re-run debounce policy; both are reasonable but extend the spec's surface. Revisit post-v0.

**Q: Why is the help long form richer than the short form?**
A: spec 0032 D-S39 / D-S40 split `-h` from `--help`. `-h` gives the working programmer the synopsis, args, flags, and exit codes. `--help` adds a Mental Model paragraph and a "see also" block. The split mirrors `man <verb>` (long, prose) vs. `<verb> -h` (short, reference). The pattern gives the experienced user a fast path and the new user a learning path without forcing a tradeoff.

**Q: Why does `completions` walk the runtime command tree instead of emitting a static script?**
A: spec 0032 D-S34 commits to runtime walking. A static script would omit user-defined commands added to a project's CLI; runtime walking includes them. The cost is per-shell template code in `cmd/glacier/internal/completions/{bash,zsh,fish,pwsh}/`, but the gain is that the generated script reflects what the user is actually running.

**Q: Why does `explain` ship pre-rendered topic files instead of rendering at runtime from spec sections?**
A: spec 0032 ships `cmd/glacier/internal/explain/topics/<slug>.md` as `embed.FS`. Runtime rendering would require shipping the spec markdown plus a markdown renderer in the binary. Pre-rendering at build time means: no markdown library in the runtime binary; no spec text shipped to users; CI byte-equality check ensures freshness. The cost is one extra build-time tool (`explaingen`); the gain is binary-size and supply-chain savings.

**Q: Why isn't `vibe` available on Windows?**
A: It is. spec 0032 covers windows/amd64 in the build matrix (D-S29) and the raw-mode discipline uses `golang.org/x/sys/windows` (already approved per spec 0002 D4). Some box-drawing characters render differently across Windows terminals; the kaomoji-only `--ascii` fallback covers the worst-case rendering.

**Q: How do I file a UX gap I notice?**
A: Open a spec amendment against 0032 with a concrete example, the surveyed-tool peer (citing this artifact's section), and an "if I were doing this for Glacier" paragraph. Octopus's review surface is exactly this.

**Q: Why is the version-check cache file format JSON instead of a binary format?**
A: spec 0032 ships `versioncheck.json` as text JSON. The file is small (a few hundred bytes), inspected rarely, and a JSON-on-disk format means users can `cat ~/.cache/glacier/versioncheck.json` to debug. Binary formats (gob, msgpack) save no measurable size at this scale and lose human-readability. The cross-tool peer is gh, which uses YAML for its update-check cache for the same reason.

**Q: What if a project's developer wants to disable `version --check` permanently?**
A: Two paths. Set `versioncheck.enabled=false` in the SDK config (per spec 0032 D-S21). Or set `GLACIER__VERSIONCHECK__ENABLED=false` in the env. Both reach the same key through the layered-config rule. The pattern matches `GH_NO_UPDATE_NOTIFIER` in spirit but uses Glacier's standard `GLACIER__*` env var convention.

**Q: Do the cross-tool patterns reflect any 2026 changes?**
A: The survey was authored 2026-05-03. Recent changes worth noting: cargo 1.85 ships rust-edition 2024 as the new `--edition` default; bun 1.2 added `--workspaces` filtering; deno 2.1 reworked the `deno doc` HTML output; golangci-lint 1.62 made SARIF first-class. No surveyed tool has materially changed its `init` / `new` / `completions` UX in the last 18 months; those conventions are stable enough to plan against.

**Q: Why does this artifact not cover Cobra directly?**
A: Cobra is a CLI library, not a CLI tool. The Glacier framework's `cli` package and `glaciergen` codegen replace what Cobra would provide. Tools built with Cobra (kubectl, gh, hugo, helm, golangci-lint) are surveyed here under their own names and inherit Cobra's conventions; this artifact captures those conventions empirically rather than re-deriving them from Cobra's API. Citation: Cobra v1.8 docs (cobra.dev) confirm the `<tool> completion <shell>` pattern as the library default.

**Q: Why is the `palette.override` config key documented as an `object` rather than a flag?**
A: spec 0032 D-S21 puts `palette.override` in the file-config tier with no flag form. The reasoning: a user who wants to override the polar-bear-blue token to a brand color does so once, in their config file, not every invocation. The cross-tool peer is `LS_COLORS` (env-var-based palette) and ripgrep's config file (file-based palette); both treat color overrides as set-once configuration rather than per-invocation flags.
